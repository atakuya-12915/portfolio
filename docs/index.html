プロジェクト① 総務チャットBOT開発

期間 2023年4月 〜 2024年3月（1年間）
技術 Power Virtual Agents / Power Automate / ChatGPT API
成果 月50時間の工数削減

🎯 課題：総務部への定型問い合わせ（会議室予約、規程確認、システム操作方法など）が月間200件以上発生し、1件あたり平均15分の対応で月50時間の工数を圧迫

🚀 実施内容・成果の詳細はこちら ▼

🚀 実施内容

  1. 要件定義：従業員30名へのヒアリングで頻出FAQ200項目を洗い出し
    • 「会議室の予約方法がわからない」「経費精算のルールを知りたい」など
  2. 設計・開発：Microsoft Teams上で動作する「総務BOT」を独学で開発
    • Power Virtual Agentsでチャットボット構築（200のFAQシナリオ実装）
    • Power AutomateでSharePointのFAQデータベースと連携
    • ChatGPT APIを活用し、曖昧な質問にも柔軟に対応（精度85%達成）
  3. 横展開・教育：グループ4社への導入支援
    • 各社の総務担当者向けに操作マニュアル作成・説明会を実施
    • FAQ項目を各社ごとにカスタマイズして横展開

✨ 成果

  • ✅ 定型問い合わせの約80%を自動対応 → 月50時間の工数削減
  • ✅ 利用率95%達成（導入3ヶ月後）
  • ✅ 改善事例発表会で最優秀部門賞受賞
  • ✅ 第69期個人表彰で社長賞（MVP）獲得

プロジェクト② 稟議申請の電子化

期間 2024年4月 〜 2025年4月（1年間）
技術 Power Apps / Power Automate / SharePoint
成果 月20時間の工数削減

🎯 課題：紙の稟議書が承認まで平均2週間かかり、進捗が見えず、承認者不在時は業務が停滞。年間約300件の稟議処理で月20時間の工数を浪費

🚀 実施内容・成果の詳細はこちら ▼

🚀 実施内容

  1. 要件定義：稟議申請〜承認までのフローを可視化
    • 申請者・承認者（課長→部長→役員）の3段階承認フローを設計
    • 金額に応じた承認ルート自動分岐（10万円未満は部長まで、以上は役員承認）
  2. 設計・開発：Power Appsで稟議申請アプリを開発
    • モバイル対応のUI設計（外出先でも承認可能）
    • Power AutomateでTeams通知・メール通知を自動化
    • SharePointに承認履歴を自動保存（監査対応）
  3. スモールスタート：総務部で1ヶ月試験運用 → 全社展開
    • フィードバックを元に「承認コメント機能」「差し戻し機能」を追加

✨ 成果

  • ✅ 承認期間を平均2週間 → 3日に短縮（約80%短縮）
  • ✅ 月20時間の工数削減（申請書作成・承認作業・進捗確認の効率化）
  • ✅ 承認状況のリアルタイム可視化で進捗確認の問い合わせがゼロに
  • ✅ 紙の稟議書をゼロ化し、ペーパーレス化に貢献

💡 学んだこと

  • スモールスタートの重要性：いきなり全社展開せず、まず小規模で検証
  • ユーザー教育の徹底：非IT人材にも分かる操作マニュアル・説明会を実施
  • 承認者の負担軽減：モバイル対応でスキマ時間に承認できる設計

📌 グループ4社全体での効果：上記は自社での実績。グループ4社全体では約4倍の効果（月280時間削減）を創出

強み① 高速PDCAで成果を量産

製造業で7年間、業務改善のPDCAを20件以上回しました。「70点で素早くリリース→横展開」を実践し、限られた予算・人員で外部ベンダーに頼らず内製化を推進。月40時間の工数削減を実現しました。

📌 具体例：稟議申請の電子化プロジェクト ▼

課題：紙の稟議書が承認まで2週間かかり、進捗が見えない
アプローチ：

  1. スモールスタート：総務部のみで1ヶ月試験運用
  2. フィードバック収集：週1回の定例で改善点を洗い出し
  3. 横展開：成功パターンをテンプレート化し、他部門へ展開

結果：承認期間を2週間→3日に短縮。月40時間の工数削減を実現

💡 学んだこと

完璧を目指さず「70点で素早くリリース」することで、失敗リスクを最小化しながら成果を積み上げる。この経験は、アジャイル開発にも通じると考えています。

強み② 多様なステークホルダーを巻き込む調整力

生産管理では「納期優先の営業・品質重視の現場・コスト重視の外注先」、総務では「数字で判断する経営層・使いやすさ重視の現場」といった異なる価値観を理解し、最適解を導きました。

📌 具体例：グループ4社横断DXプロジェクト ▼

状況：各社で異なるツールを使用しており、情報共有が非効率
アプローチ：

  1. 各社の要望をヒアリング：「使いやすさ重視」「コスト最小化」など
  2. 共通課題を抽出：全社で使える機能を優先順位付け
  3. 段階的導入：まず1社で成功させてから横展開

結果：グループ4社で統一ツールを導入。情報共有が円滑化

💡 学んだこと

異なる立場の人々の「本当のニーズ」を理解し、Win-Winの提案をすることで、合意形成がスムーズになる。この経験は、開発現場での要件定義でも活きると考えています。

強み③ 自走力と技術への探究心

大学院では成績1位で卒業し国際学会で発表。DX推進では誰も教えてくれない環境で公式ドキュメント・Udemy・書籍・YouTubeなどを活用し自己学習。スクールの24週Javaコースを18週で完走し、現在もReact開発を継続中。

  1. 公式ドキュメントを読む：ChatGPT APIの仕様を理解

💡 学んだこと

NAGOYAMESHI

食べログ風レストラン検索・レビュー投稿アプリ

📸 画面イメージ

📝 概要

「ユーザーが飲食店を探しやすく、信頼できるレビューを投稿できる仕組み」を目指して開発しました。

🛠️ 使用技術

  • Java 21
  • Spring Boot 3.5.4
  • MySQL 8.0
  • Thymeleaf
  • Spring Security
  • stripe API 28.2.0

✨ 実装機能

  • 会員登録・ログイン認証（一般/管理者）
  • レストラン検索（エリア・価格帯・評価での絞り込み）
  • レビュー投稿・編集・削除機能
  • お気に入り登録（多対多のリレーション設計）

🎯 なぜこのアプリを作ったのか

スクールの課題として開発しましたが、「ただ作る」のではなく、「実務レベルの品質」を目指しました。特に、Spring Securityでのロール別アクセス制御は、実務でも使える重要な機能だと考え、徹底的に実装しました。

💪 苦労した点・工夫した点

  • MyBatisでの複雑なJOIN処理：レストラン・レビュー・ユーザーの3テーブルを結合する際、N+1問題を回避するためクエリを最適化
  • エラーハンドリング：ユーザーが入力ミスをしても、分かりやすいエラーメッセージを表示
  • レスポンシブ対応：PCだけでなく、スマホでも使いやすいUI設計

📈 このアプリで学んだこと

「ユーザーが使いやすいUI」と「保守しやすいコード」を両立させる難しさ。特に、Spring Securityの実装では、公式ドキュメントを何度も読み返し、理解を深めました。
